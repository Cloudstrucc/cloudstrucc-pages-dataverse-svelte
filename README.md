# Cloudstrucc Pages Studio for Dataverse

This repository bootstraps the Cloudstrucc Pages Studio schema and Studio web resources in a Dataverse development environment, then exports authoritative managed and unmanaged solution packages generated by Dataverse.

## Important: use the bootstrap workflow

Do not import the old hand-authored `solution/full/packed` or `solution/schema/packed` packages from versions 1.0.2 and earlier. Those packages were prototypes and are not authoritative Dataverse exports.

Version **1.0.5.0** creates the tables through the Dataverse metadata Web API. Each table is created with its `cs_Name` primary-name column and all additional columns in the same `EntityDefinitions` request. Microsoft requires the primary-name definition to be included when the table is created and returns the table metadata URI in the `OData-EntityId` response header. The bootstrap uses that metadata ID for all follow-up work and resolves existing tables through a filtered `EntityDefinitions` collection query instead of the unreliable `LogicalName` alternate-key route.

## Prerequisites

- macOS, Linux, or Windows
- Python 3.10 or later
- Node.js 18 or later and npm (only required for the Playwright UI test suite, see `docs/TESTING.md`)
- Azure CLI (`az`)
- Microsoft Power Platform CLI (`pac`)
- Dataverse System Administrator or System Customizer privileges
- Permission to create tables, solutions, web resources, SiteMaps, and model-driven apps

Authenticate first:

```bash
az login
pac auth create --environment "https://goc-theme-dev.crm3.dynamics.com"
```

## First installation in the development environment

```bash
chmod +x scripts/*.sh

python3 scripts/validate-schema.py
python3 -m unittest discover -s tests -v

./scripts/first-install.sh \
  --environment-url "https://goc-theme-dev.crm3.dynamics.com" \
  --managed
```

The operation is idempotent. If an earlier run created `cs_website` and then stopped, run the same command again. The bootstrap publishes pending customizations, locates the existing table, verifies its primary-name column, creates only missing columns, continues with the remaining tables and web resources, and finally exports the solutions.

## What the first-install command does

1. Creates or reuses the `Cloudstrucc` publisher.
2. Creates or reuses these unmanaged development solutions:
   - `CloudstruccPagesSchema`
   - `CloudstruccPagesStudio`
3. Publishes pending metadata from any interrupted earlier run.
4. Creates or repairs all 12 Cloudstrucc tables.
5. Creates or updates the admin and Studio web resources (including the admin app icon).
6. Sets the admin HTML web resource as the full solution configuration page.
7. Creates or updates the `Cloudstrucc Pages Admin Sitemap` SiteMap, pointing its subarea at the admin web resource via a `$webresource:` directive.
8. Creates or updates the `Cloudstrucc Pages Admin` model-driven app (AppModule), attaches the SiteMap and admin web resource to it via `AddAppComponents`, runs `ValidateApp`, and publishes the app.
9. Publishes all customizations.
10. Exports Dataverse-generated solution ZIPs.

## Accessing the admin console as a model-driven app

Earlier versions only exposed the admin web resource as a raw HTML file with no app shell around it. As of this bootstrap, the admin console is reachable the normal Dataverse way:

1. In the target environment, open **Apps** (make.powerapps.com or the classic app list).
2. Launch **Cloudstrucc Pages Admin**.
3. The SiteMap opens directly to the **Admin Console** subarea, which renders `admin.html`.

This pass wires up the app/form/SiteMap shell only. The admin console's "Create website" flow currently updates its own in-memory demo state client-side and does not yet write to Dataverse — `src/shared/js/dataverse-client.js` and `src/shared/js/provisioner.js` contain a working `cs_website` draft-creation client but are not referenced by `admin.html` yet. Wiring the UI to those files, and the governed provisioning flow or custom API that turns a draft into a live Power Pages site (`docs/POWER_PAGES_PROVISIONING.md`), is separate, later work.

## Expected exported packages

After a successful run:

```text
solution/exported/schema/CloudstruccPagesSchema_1_0_5_0_unmanaged.zip
solution/exported/schema/CloudstruccPagesSchema_1_0_5_0_managed.zip
solution/exported/full/CloudstruccPagesStudio_1_0_5_0_unmanaged.zip
solution/exported/full/CloudstruccPagesStudio_1_0_5_0_managed.zip
```

The export files do not exist until the bootstrap completes. A `solution file not found` error after a failed bootstrap is therefore expected; fix or rerun the bootstrap first.

## Install in another Dataverse environment

For production or another controlled environment, import the full managed package only:

```bash
./scripts/import-solution.sh \
  --environment-url "https://YOUR-TARGET.crm3.dynamics.com" \
  --solution-path \
  "solution/exported/full/CloudstruccPagesStudio_1_0_5_0_managed.zip"
```

For another development environment, use the full unmanaged package:

```bash
./scripts/import-solution.sh \
  --environment-url "https://YOUR-DEV.crm3.dynamics.com" \
  --solution-path \
  "solution/exported/full/CloudstruccPagesStudio_1_0_5_0_unmanaged.zip"
```

Replace `YOUR-TARGET` or `YOUR-DEV` with a real environment URL. The literal URL `https://TARGET.crm3.dynamics.com` is only a placeholder.

## Optional split import

The full solution contains the schema and Studio artifacts, so one full-solution import is preferred. For a split deployment:

```bash
./scripts/import-solution.sh \
  --environment-url "https://YOUR-TARGET.crm3.dynamics.com" \
  --solution-path \
  "solution/exported/schema/CloudstruccPagesSchema_1_0_5_0_managed.zip"

./scripts/import-solution.sh \
  --environment-url "https://YOUR-TARGET.crm3.dynamics.com" \
  --solution-path \
  "solution/exported/full/CloudstruccPagesStudio_1_0_5_0_managed.zip"
```

## Recovery from the `cs_website` propagation error

Earlier code created `cs_website`, slept briefly, and immediately queried:

```text
EntityDefinitions(LogicalName='cs_website')/Attributes
```

Some environments had not yet made the alternate key available and returned 404 even though the table had been created. Version 1.0.5.0 fixes this by:

- creating all columns in the initial table request;
- capturing the `OData-EntityId` metadata GUID;
- retrieving new tables by metadata GUID;
- resolving existing tables through `EntityDefinitions?$filter=LogicalName eq ...`;
- using the metadata GUID for every attribute lookup and create operation;
- retrying metadata availability for up to two minutes;
- publishing pending customizations before resuming;
- recovering from a duplicate-create response when an interrupted run already created the table.

Rerun:

```bash
./scripts/first-install.sh \
  --environment-url "https://goc-theme-dev.crm3.dynamics.com" \
  --managed
```

Do not run the target-environment import command until the four exported ZIPs exist.
